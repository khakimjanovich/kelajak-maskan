# Wants To Plans

## Overview

Turn wants, needs, and desired outcomes into plans that can be understood, validated, and handed off to execution.

## Use This Skill When

- A desire or need is clear, but the path is not
- A user or client request must be translated into builder responsibilities
- A builder or admin goal must be translated into user value and plan requirements
- A vague idea needs scope, priorities, constraints, and sequencing

## Required Workflow

1. Identify the want, need, concern, or desired outcome.
2. Extract constraints, risks, resources, time, and non-negotiables.
3. Translate the want into a draft plan: objective, scope, priorities, responsibilities, safeguards, and sequence.
4. Run the draft plan through the gates below.
5. For `facts` and `experience`, query trustworthy resources first. Only ask the prompter if source access, source pointers, or critical context is missing.
6. If a gate fails because information is missing, stop and name what is missing before continuing.
7. If the draft fails the `facts`, `constraints`, or `experience` gate, revise the plan or clarify the want, then repeat the loop only after that gate is addressed.
8. If the draft fails the `ikhlas` gate, stop planning the want. Do not continue the loop until the want is abandoned, corrected, or purified.
9. Restate the grounded plan in clear language.
10. Stop at plan quality. Do not execute yet.

## Validation Criteria

Validate each draft plan against:

- facts
- constraints
- experience
- `ikhlas` (`iklas`): in the meaning of Surah Al-Ikhlas from the Holy Quran, purity and sincerity of devotion to Allah alone, with faithfulness and unmixed intention

## Facts Gate

Validate the draft against known reality.

- Query trustworthy resources first. Do not let model inference decide what is true.
- Check claims, assumptions, and dependencies against source-backed facts.
- If a required fact is false, unverified, or missing, stop and name the exact fact gap or missing source access.
- Ask the prompter only for missing source pointers, permissions, or context needed to complete validation.
- Do not treat guesses as validated plan inputs.

## Constraints Gate

Validate the draft against actual limits.

- Check time, resources, access, permissions, obligations, and non-negotiables.
- If the plan does not fit the constraints, stop and name the exact limit that blocks it.
- Only continue after the plan is reduced, resourced, or corrected.

## Experience Gate

Validate the draft against prior attempts and learned reality.

- Query trustworthy experience sources first, such as documented history, logs, reports, or verified records of prior outcomes.
- Check whether past experience shows this path repeatedly fails or carries known risks.
- If source-backed experience conflicts with the draft, stop and name what must change materially.
- Ask the prompter only when source access, source pointers, or critical context is missing.
- Do not continue with the same failed pattern under a new description.

## Ikhlas Gate

Validate the want and the plan against `ikhlas` as a whole, not as a decorative value.

- Check the motive, not just the method.
- Reject wants driven by vanity, showing off, ego, or collision with devotion to Allah.
- If the want itself conflicts with this meaning, stop thinking about that want instead of refining the plan around it.

## Translation Rule

- If the speaker is expressing a user need, translate that need into builder responsibilities and safeguards.
- If the speaker is expressing a builder goal, translate that goal into user impact, trust, and plan requirements.

## Good Outcomes

- Wants become structured plans
- Plans are validated before execution
- Failed gates become explicit
- Needs become responsibilities
- Builder intent becomes understandable
- Planning is explicit before action starts

## Avoid

- Treating a want as if it were already a plan
- Mistaking a preferred solution for the real need
- Treating an unchecked draft as ready for action
- Continuing past a failed or unknown gate
- Letting AI decide facts or past outcomes without source-backed validation
- Jumping from desire straight into execution
- Hiding responsibilities inside vague language
